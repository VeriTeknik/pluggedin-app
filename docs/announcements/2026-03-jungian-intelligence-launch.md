3. **Archetype Router** — Deterministic context-to-archetype mapping (no LLM call). Four archetypes (Shadow/Sage/Hero/Trickster) route patterns based on observation type, outcome, error presence, and consecutive failure count. Wraps the existing injection engine without modifying it.

4. **Individuation Service** — 4-component maturity score (0-100): Memory Depth, Learning Velocity, Collective Contribution, Self-Awareness. Five levels: Nascent → Individuated. All computed from existing data — no new tracking.

**Privacy model:** HMAC-SHA256 profile hashing (irreversible). k-anonymity (patterns invisible until ≥3 unique profiles contribute). Temporal aggregation (day-of-week/hour-of-day only, no exact timestamps in collective layer). Content normalization strips identifiers before pattern storage.

**Title:** We implemented Jung's collective unconscious as an AI memory system — here's how it works

**Text:**

I've been working on Plugged.in, an open-source AI infrastructure platform. Our memory system already had concentric rings (inspired by human cognition — fresh observations decay through compression stages over time). But every agent was an island. Thousands of agents hitting the same errors, discovering the same patterns, independently.

So we asked: what if we could build a collective unconscious for AI?

Carl Jung's model turned out to be surprisingly implementable:

**1. Synchronicity Detection**

Not the mystical kind. SQL-based temporal co-occurrence analysis across anonymized events. The detector runs periodically and looks for:
- **Co-occurrences**: tools used together within a 5-minute window across ≥3 profiles
- **Failure correlations**: error rate spikes on specific days/hours
- **Emergent workflows**: 3-tool sequences that multiple profiles independently discover

The best example: it found that deploy failures are 3.4x more common on Friday afternoons. 11 profiles, nobody told it to look. When someone deploys on Friday at 2 PM, the system automatically warns them.

**2. Dream Processing**

During low-activity periods, the system does memory consolidation — literally the biological function of sleep.

It uses vector embeddings to cluster semantically similar memories (Union-Find algorithm, cosine similarity ≥ 0.75). When a cluster forms, an LLM synthesizes the members into one authoritative entry.

Real numbers: 15 developers documented the same Docker permission error independently. 15 × 500 tokens = 7,500 tokens of overlapping knowledge. After dream processing: 1 comprehensive guide, 600 tokens. 92% reduction, higher quality.

**3. Archetype Router**

Four archetypes classify every pattern:
- **Shadow**: anti-patterns, security warnings, "don't do this"
- **Sage**: best practices, solutions, "here's the right way"
- **Hero**: complete workflows, "follow these steps"
- **Trickster**: edge cases, gotchas, "this will bite you"

The routing is **deterministic** — pure function of context (observation type, outcome, error presence). No LLM call. Wraps the existing injection engine.

**4. Individuation Score**

0-100 maturity score across four dimensions:
- Memory Depth (how rich/diverse the memory store is)
- Learning Velocity (rate of knowledge acquisition)
- Collective Contribution (net giver vs taker)
- Self-Awareness (how well memories are consolidated)

Five levels: Nascent → Developing → Established → Mature → Individuated

A new team member's AI starts at Nascent but immediately inherits collective patterns. By week 2, measurably outperforms what months of solo experience would produce.

**Privacy**

This was the hardest part. Collective learning is a privacy minefield.

Our approach is structural, not policy-based:
- Profile identity: HMAC-SHA256 one-way hash (can't reverse it)
- k-anonymity: patterns only surface when ≥3 unique profiles contribute
- Temporal data: aggregated to day-of-week/hour-of-day (no exact timestamps)
- Content: normalized to strip identifiers before collective storage

You learn from everyone. No one can learn about you.

**Stack:** Next.js 15, PostgreSQL with partitioned tables, Drizzle ORM, zvec for vector search. 8 new files, 3 new tables, ~106 lines changed in existing code.
